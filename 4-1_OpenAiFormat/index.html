    OpenAI JSON Format Specification

    Overview

    This document provides detailed specifications for the JSON request and response formats used when interacting with OpenAI-compatible APIs. The OpenAI format is an industry standard adopted by many LLM service providers.

    Core Features: - Standardized message format - Support for function calling - Use max_tokens to control output length - Support for tool_choice parameter to control tool usage strategy - Response format conforms to OpenAI API specification

    Request Format

    Basic Request Format

    The OpenAI request body contains the following main fields:

    • messages: Message array containing conversation history
    • model: Model name (e.g., "gpt-4o-mini")
    • max_tokens: Maximum number of tokens to generate
    • temperature: Temperature parameter
    • top_p: Top-p sampling parameter
    • stream: Whether to stream responses
    • tools: Tool definition array (optional)
    • tool_choice: Tool selection strategy (optional)
    • chat_template_kwargs: Chat template parameters (optional)

    Basic Request Example

    {
  "chat_template_kwargs": {
    "enable_thinking": false
  },
  "max_tokens": 4096,
  "messages": [
    {
      "content": "You are a helpful assistant with multiple tools. When answering user questions, please choose and invoke tools appropriately based on the content of the question, and clearly explain the purpose and results of the tool after invocation. Current time: 2025-12-11 17:36:42",
      "role": "system"
    },
    {
      "content": "tell me one news from San Francisco today.",
      "role": "user"
    }
  ],
  "model": "gpt-4o-mini",
  "stream": false,
  "temperature": 0.7,
  "tool_choice": "auto",
  "tools": [
    {
      "function": {
        "description": "Search the web using keywords to retrieve general information that does not change over time. Supports filtering by country and other conditions to get web content, titles, and link information",
        "name": "search_web2",
        "parameters": {
          "properties": {
            "country": {
              "description": "Country name (must be in English), used to limit search results from a specific country",
              "enum": ["china", "united states", "zimbabwe"],
              "type": "string"
            },
            "query": {
              "description": "Search query string for searching relevant information on the web. Keyword combinations should be concise and clear, avoiding redundant information while ensuring they accurately reflect the core needs of the user's question. Keyword combinations should conform to search engine syntax and logical rules",
              "minLength": 1,
              "type": "string"
            }
          },
          "required": ["query"],
          "type": "object"
        }
      },
      "type": "function"
    }
  ],
  "top_p": 0.95
}

    Request Format with Tool Calls

    When the model needs to call a tool, the request body will include the assistant's tool call message and the tool's execution result:

    {
  "chat_template_kwargs": {
    "enable_thinking": false
  },
  "max_tokens": 4096,
  "messages": [
    {
      "content": "You are a helpful assistant with multiple tools. When answering user questions, please choose and invoke tools appropriately based on the content of the question, and clearly explain the purpose and results of the tool after invocation. Current time: 2025-12-11 17:36:42",
      "role": "system"
    },
    {
      "content": "tell me one news from San Francisco today.",
      "role": "user"
    },
    {
      "content": "",
      "role": "assistant",
      "tool_calls": [
        {
          "function": {
            "arguments": "{\"country\":\"united states\",\"query\":\"San Francisco news December 11 2025\"}",
            "name": "search_web2"
          },
          "id": "call_hMffYLcob2f6Q3FIxllY8xuy",
          "type": "function"
        }
      ]
    },
    {
      "content": "{\"msg\":\"\",\"responses\":[{\"score\":0.8020137,\"snippet\":\"Thursday Dec 11, 2025 9:00 AM - 12:00 PM PST. Location. Center for Architecture + Design 140 Sutter Street San Francisco, CA 94104. Fees/Admission. $10.00\",\"title\":\"BuildSF Summit: The Next Chapter for Downtown ...\",\"url\":\"https://business.sfchamber.com/events/details/buildsf-summit-the-next-chapter-for-downtown-san-francisco-12899\"},...],\"status\":\"success\"}",
      "role": "tool",
      "tool_call_id": "call_hMffYLcob2f6Q3FIxllY8xuy",
      "tool_name": "search_web2"
    }
  ],
  "model": "gpt-4o-mini",
  "stream": false,
  "temperature": 0.7,
  "tool_choice": "auto",
  "tools": [...],
  "top_p": 0.95
}

    Response Format

    Basic Response Format

    The OpenAI response contains the following fields:

    • code: HTTP status code
    • raw_data: Raw response data (conforms to OpenAI API specification)
    • role: Message role
    • tool_calls: Tool call array (if present)

    Key Field Descriptions

    Request Fields

    • messages: Message array supporting system, user, assistant, and tool roles
    • model: Model identifier
    • max_tokens: Maximum number of tokens to generate (replaces Ollama's options.num_predict)
    • temperature: Temperature parameter (0.0-2.0)
    • top_p: Top-p sampling parameter (0.0-1.0)
    • stream: Whether to stream responses (true/false)
    • tools: Tool definition array, each tool contains function and type fields
    • tool_choice: Tool selection strategy
    • "auto": Automatically decide whether to call tools
    • "none": Do not call tools
    • "required": Must call tools
    • {"type": "function", "function": {"name": "tool_name"}}: Specify a particular tool to call
    • chat_template_kwargs: Chat template parameters, such as enable_thinking to control whether to enable thinking process

    Response Fields

    • code: HTTP status code
    • raw_data: Response data conforming to OpenAI API specification
    • choices: Choice array containing generated responses
      • finish_reason: Completion reason ("stop", "tool_calls", "length", etc.)
      • message: Message content
      • content: Text content (may be null)
      • role: Message role
      • tool_calls: Tool call array
      • annotations: Annotation array
      • refusal: Refusal content (if any)
    • created: Creation timestamp (Unix timestamp)
    • id: Response ID
    • model: Model used
    • object: Object type ("chat.completion")
    • usage: Token usage statistics
      • prompt_tokens: Number of prompt tokens
      • completion_tokens: Number of completion tokens
      • total_tokens: Total number of tokens
      • prompt_tokens_details: Detailed prompt token information
      • completion_tokens_details: Detailed completion token information
    • role: Message role
    • tool_calls: Tool call array (if the model decides to call tools)

    Tool Calling Workflow

    1. Initial Request: Send user message and tool definitions, set tool_choice to "auto"
    2. Tool Call Response: Model returns tool_calls containing tools to call and parameters
    3. Tool Execution: Execute tools and obtain results
    4. Subsequent Request: Add tool execution results as tool role messages to message history
    5. Final Response: Model generates final answer based on tool results

    Key Differences from Ollama Format

    1. Parameter Configuration: OpenAI uses top-level fields (max_tokens, temperature), while Ollama uses an options object
    2. Tool Selection: OpenAI supports tool_choice parameter, providing more flexible tool call control
    3. Response Structure: OpenAI responses conform to standard API specification, containing choices array and usage statistics
    4. Thinking Process: OpenAI controls through chat_template_kwargs.enable_thinking, while Ollama directly includes thinking field in responses
    5. Time Statistics: OpenAI provides token usage statistics, while Ollama provides detailed performance time statistics
    6. Tool Parameter Format: OpenAI tool call arguments are always JSON strings

    Notes

    1. Tool Parameters: Tool call arguments must be in JSON string format
    2. finish_reason: When the model decides to call tools, finish_reason is "tool_calls"
    3. content Field: When the model calls tools, content may be null
    4. tool_choice: Proper use of tool_choice can control model behavior and avoid unnecessary tool calls
    5. Token Statistics: The usage field provides detailed token usage information, useful for cost calculation and optimization
